## 1. 프로젝트 개요

### 1.1 한 줄 정의

**오픈런 트래픽을 견디는 단일 구단 티켓팅 백엔드 시스템.**

### 1.2 만드는 이유

KBO 리그의 특정 카드 — 한국시리즈, 신축 구장 개막전, 라이벌 매치 — 는 티켓 오픈 직후 1분 안에 수만 명의 동시 접속이 몰린다. 매년 인터파크·티켓링크 같은 기존 예매 플랫폼이 *대기열 폭주, 좌석 중복 점유, 결제 실패 후 좌석 미반환* 같은 장애를 반복해서 뉴스에 오른다.

본 프로젝트는 이러한 **고트래픽·고정합성이 동시에 요구되는 도메인**을 직접 구현하면서, 분산 시스템의 핵심 패턴(샤딩, 분산락, 비동기 메시징, Transactional Outbox, CQRS)이 *실제로 어떤 문제를 풀기 위해 존재하는지*를 코드와 실험으로 검증하는 것을 목표로 한다.

### 1.3 도메인 컨텍스트: 왜 한화 이글스 / 대전인가

- 2025년 신축 개장한 대전 신구장은 **시즌 홈경기 전 좌석 매진**이라는 KBO 사상 유례없는 흥행을 기록했다.
- 즉, "오픈런 트래픽"이라는 본 프로젝트의 핵심 가정이 *현실에서 가장 첨예하게 발생하는 구단*이다.
- 신축 단일 사용 구장이므로 좌석 체계가 단순하고, 도메인 모델링 시 불필요한 복잡도를 피할 수 있다.

> ⚠️ 본 프로젝트는 학습 및 포트폴리오 목적의 시뮬레이션이며, 실제 구단·구장과 어떠한 제휴나 데이터 연동도 없다. 좌석 배치 등은 공개된 정보를 모티브로 한 가상 모델이다.

## 2. 문제 정의

### 2.1 기존 티켓팅 플랫폼의 페인포인트

| # | 문제 | 빈도 | 본 프로젝트가 다루는가 |
|---|---|---|---|
| 1 | 오픈 직후 서버 응답 불가 (트래픽 폭주) | 매 인기 매치 | ✅ |
| 2 | 동일 좌석이 두 명에게 동시 점유됨 | 종종 발생 | ✅ |
| 3 | 결제는 성공했는데 좌석은 미확정 (정합성 깨짐) | 드물지만 치명적 | ✅ |
| 4 | 대기열 새로고침 시 순번이 뒤로 밀림 (공정성 훼손) | 자주 | ✅ |
| 5 | 결제 페이지 진입 후 무한 대기 → 시간 초과 | 종종 | ✅ |
| 6 | 모바일 앱 UX 불편 | 항상 | ❌ (스코프 외) |
| 7 | PG사 장애 시 전체 시스템 마비 | 드물게 | ✅ (장애 격리로 다룸) |

### 2.2 본질적 기술 문제로 환원

위 페인포인트들은 결국 다음 4가지 분산 시스템 문제로 환원된다:

1. **읽기·쓰기 트래픽 비대칭** — 좌석 조회는 초당 수만, 좌석 점유는 초당 수천. 같은 모델로 처리하면 한쪽이 다른 쪽을 죽인다.
2. **동시성 정합성** — 단일 자원(좌석)에 대한 동시 점유 요청을 *정확히 한 명에게만* 허용해야 한다.
3. **분산 트랜잭션** — 결제(외부 시스템) ↔ 좌석 확정(내부 DB) ↔ 알림(비동기) 의 원자성을 보장해야 한다.
4. **공정성** — 먼저 줄 선 사용자가 먼저 처리되어야 한다. FIFO 보장.

**이 4가지가 본 프로젝트가 풀고자 하는 진짜 문제다.** 기능 명세는 이 문제들을 드러내기 위한 수단이다.

## 3. 목표와 비목표

### 3.1 목표 (In-Scope)

- 단일 구단·단일 구장 기준의 **티켓 예매 백엔드 API** 구현
- 오픈런 시점의 **고동시성 좌석 점유** 정확성 보장
- 대기열 시스템을 통한 **공정한 진입 순서** 보장
- 결제-예매 간 **데이터 정합성** 보장 (결제 성공 = 좌석 확정 보장)
- 장애 격리: 알림·통계 등 부가 시스템 장애가 핵심 예매 플로우를 막지 않음
- AWS 기반 **운영 환경 배포** 및 부하 테스트 검증

### 3.2 비목표 (Out-of-Scope)

명시적으로 다루지 *않는* 영역. 기획 단계에서 빠져있는 것이 아니라, **의도적으로 제외**한 것이다.

- ❌ **프론트엔드**: API 검증은 Postman/Swagger로 대체. UI는 만들지 않음.
- ❌ **확장 인증 기능**: 기본 회원가입·로그인·JWT는 P0에 포함하되, 다음은 의도적으로 제외 — Refresh token rotation, 이메일/본인인증, 비밀번호 재설정, 소셜 로그인(카카오·네이버 등), 로그아웃 토큰 블랙리스트, 계정 잠금. 본 프로젝트의 학습 목표는 분산 트랜잭션과 동시성이며, 인증은 *동작하는 수준*까지만 구현한다.
- ❌ **실제 PG 연동**: 결제는 시뮬레이터로 대체. 단, *실제 PG의 동작 특성*(지연, 멱등성, 콜백, 실패 케이스)을 모방한다.
- ❌ **환불 정책·고객센터 등 운영 기능**: 도메인 복잡도만 증가시키고 기술 학습엔 도움 안 됨.
- ❌ **모바일 푸시·이메일 발송 실제 연동**: Kafka 토픽까지만 발행하고, 컨슈머는 로깅으로 대체.
- ❌ **다구단 통합**: 단일 구단으로 충분히 핵심 문제를 다룰 수 있음. 통합은 비즈니스 이유가 약함.

## 4. 타겟 사용자 & 핵심 시나리오

### 4.1 타겟 사용자

| 페르소나 | 설명 | 핵심 니즈 |
|---|---|---|
| **김덕후** (열성팬) | 한화 시즌권 노리는 30대 직장인. 티켓 오픈 시간에 알람 맞춤. | 빠르게, 공정하게, 원하는 좌석을 확보 |
| **박입문** (입문 팬) | 친구 따라 첫 야구장 가는 20대. 어디 앉을지 모름. | 좌석 등급 정보를 보고 결정 |

UI를 만들지 않으므로 페르소나는 *API 동작의 시나리오 검증*용으로만 쓴다.

### 4.2 핵심 시나리오

**Scenario A: 일반 시점 예매 (저트래픽)**

1. 김덕후가 다음 주 평일 경기 좌석 배치도를 조회한다.
2. 1루 외야 자유석을 선택한다.
3. 결제 페이지로 진입한다.
4. 결제를 완료한다.
5. 예매 내역을 확인한다.

**Scenario B: 오픈런 예매 (고트래픽)** — *본 프로젝트의 핵심*

1. 한국시리즈 1차전 티켓 오픈 5분 전, 5만 명이 대기 페이지에 접속한다.
2. 정각에 대기열이 열리고, 사용자별 순번이 부여된다.
3. 김덕후의 순번이 호출되어 진입 토큰을 발급받는다.
4. 김덕후는 토큰으로 좌석 선택 페이지에 진입한다.
5. 김덕후가 1루 응원석 A-3-15를 선택한다 — 동시에 같은 좌석을 선택한 다른 사용자 12명이 있다.
6. 시스템은 *정확히 한 명에게만* 점유를 허용한다. 김덕후가 성공한다.
7. 김덕후는 결제 시간(7분) 안에 결제를 완료한다.
8. 결제 성공 이벤트가 발생하면, 좌석은 영구 확정되고 알림 이벤트가 발행된다.
9. 만약 김덕후가 7분 안에 결제하지 못하면, 좌석은 자동으로 풀려서 대기 중인 다른 사용자에게 돌아간다.

**Scenario C: 결제 시스템 장애** — *정합성 검증*

1. 김덕후가 결제를 시도한다.
2. PG사 응답이 30초 동안 없다.
3. 시스템은 결제 상태를 "처리 중"으로 유지한다 (좌석은 점유 상태 유지).
4. PG 응답이 도착하면 — 성공이든 실패든 — 멱등하게 처리한다.
5. 결제 성공 시 좌석 확정 + 알림 발행이 *원자적으로* 일어난다 (Transactional Outbox).

## 5. 핵심 기능 명세

우선순위로 구분. P0 = 반드시, P1 = 가능하면, P2 = 시간 남으면.

### 5.1 P0 (필수, 미구현 시 프로젝트 실패)

| 기능 | 설명 |
|---|---|
| 회원가입 | userId/password로 사용자 생성. BCrypt 해싱. |
| 로그인 | 자격 검증 후 JWT access token 발급 (장시간 TTL, 단순화) |
| JWT 인증 필터 | Spring Security 기반. 보호된 엔드포인트 자동 검증. |
| 경기 조회 | 경기 일정·좌석 등급별 잔여 정보 조회 |
| 좌석 배치도 조회 | 특정 경기의 전체 좌석 상태(가용/점유/매진) 조회 |
| 대기열 진입 | 사용자가 대기열에 등록되고 순번을 받음 |
| 대기열 호출 | 시스템이 순번에 따라 진입 토큰 발급 |
| 좌석 점유 | 토큰을 가진 사용자가 좌석을 임시 점유 (TTL 7분) |
| 결제 요청 | 점유한 좌석에 대해 결제 시작 |
| 결제 결과 처리 | PG 콜백 수신 및 멱등 처리 |
| 예매 확정 | 결제 성공 시 좌석 영구 확정 + 후속 이벤트 발행 |
| 점유 만료 자동 해제 | 결제 시간 초과 시 좌석 자동 반환 |
| 예매 내역 조회 | 사용자의 예매 내역 조회 |
| 예매 취소 | 사용자가 본인 예매를 취소 |

### 5.2 P1 (있으면 좋음)

| 기능 | 설명 |

## 6. 비기능 요구사항 (NFR)

모든 수치는 부하 테스트로 검증 가능한 것만 적었다.

### 6.1 성능 목표

| 지표 | 목표 | 측정 방법 |
|---|---|---|
| 좌석 조회 처리량 | **3,000 req/s 이상** | k6 부하 테스트 |
| 좌석 점유 처리량 | **500 req/s 이상** | k6 부하 테스트 |
| 좌석 조회 p99 응답시간 | **200ms 이하** | k6 + 애플리케이션 메트릭 |
| 좌석 점유 p99 응답시간 | **500ms 이하** | k6 + 애플리케이션 메트릭 |
| 대기열 진입 처리량 | **5,000 req/s 이상** | k6 부하 테스트 |

> ⚠️ 위 수치는 *목표값*이다. 1개월차 시점의 추정이며, 부하 테스트 결과에 따라 v0.2에서 조정될 수 있다. 달성하지 못한 경우에도 *왜 못 달성했는지*를 분석하는 것이 학습 목표다.

### 6.2 정합성 목표

| 지표 | 목표 |
|---|---|
| 동일 좌석 중복 점유 | **0건** (절대 허용 불가) |
| 결제 성공 → 좌석 미확정 | **0건** |
| 결제 실패 → 좌석 미반환 | **0건** (TTL 보장) |
| 대기열 FIFO 위반 | **5% 이내** (네트워크 지연으로 인한 미세 역전 허용) |

### 6.3 가용성 목표

| 지표 | 목표 |
|---|---|
| 알림 시스템 장애 시 | 핵심 예매 플로우는 정상 동작 |
| Redis 단일 장애 시 | 좌석 점유는 일시 중단되나 데이터 손실 없음 |
| Kafka 장애 시 | 결제는 정상, 후속 이벤트는 복구 후 재처리 (Outbox) |

## 7. 도메인 모델 개요

상세 ERD는 별도 문서(`docs/ERD.md`)로 작성. 여기서는 핵심 개념과 관계만 정리한다.

### 7.1 핵심 엔티티

- **Stadium** — 구장. 단일 인스턴스 (대전 신구장).
- **Section** — 구역 (예: 1루 내야, 외야 자유석). 좌석 등급의 단위.
- **Seat** — 개별 좌석. Section에 속함. (구역, 열, 번호)로 식별.
- **Game** — 경기. 일자, 홈/원정, 시작 시각.
- **GameSeat** — *경기별 좌석 인스턴스*. Game × Seat의 조합. 가격, 상태(가용/점유/확정/취소)를 가짐.
- **Reservation** — 예매. 사용자가 GameSeat을 점유·확정한 단위.
- **Payment** — 결제. Reservation과 1:1.
- **WaitingEntry** — 대기열 항목. (경기, 사용자, 진입 시각, 순번).
- **OutboxEvent** — 트랜잭션 아웃박스 이벤트. 결제 확정 시 작성, 별도 워커가 Kafka로 발행.

### 7.2 핵심 설계 결정

**왜 Seat과 GameSeat을 분리하는가?**
물리 좌석(Seat)은 변하지 않지만, *경기마다 가격이 다르고 상태가 독립적*이다. 한 좌석이 여러 경기에 동시 존재해야 하므로 분리한다. 만약 분리하지 않으면 모든 좌석이 단 하나의 상태만 가지게 되어 다중 경기 운영이 불가능하다.

**왜 Reservation과 Payment를 분리하는가?**
Reservation은 "좌석을 잡았다"는 사실, Payment는 "돈이 오갔다"는 사실. 둘은 라이프사이클이 다르다 (예: 결제는 부분 환불, 재시도가 가능). 분리하지 않으면 환불·재결제 같은 미래 확장이 막힌다.

**왜 OutboxEvent 테이블이 필요한가?**
결제 성공 트랜잭션 안에서 Kafka에 직접 발행하면, *DB는 커밋됐는데 Kafka 발행이 실패*한 케이스에서 데이터가 영원히 어긋난다. Outbox 테이블에 이벤트를 같은 트랜잭션으로 기록하고 별도 워커가 Kafka로 옮기면, *최소 한 번(at-least-once)* 전달이 보장된다. 이는 분산 시스템에서 트랜잭션 경계를 다루는 표준 패턴이다.

## 8. 기술적 도전 과제

> 본 프로젝트가 *진짜로 풀고자 하는 문제들*. 면접에서 가장 깊게 이야기할 영역이다.

### 도전 1: 좌석 동시 점유

**문제**: 동일 좌석에 대한 동시 요청이 수십~수백 건 들어올 때, *정확히 한 명*에게만 점유를 허용해야 한다.

**고려한 접근**:
1. **DB 비관적 락 (`SELECT ... FOR UPDATE`)** — 단순하지만 락 경합 시 처리량 급락.
2. **DB 낙관적 락 (version 컬럼)** — 충돌 시 재시도 비용. 인기 좌석에선 대부분 실패.
3. **Redis 분산락** — 빠르지만 락 해제 누락·노드 장애 시 복잡.
4. **Redis 단일 명령 원자성 (`SET NX`)** — 락이 아닌 *상태 자체*를 Redis에 두고 원자 연산으로 점유.

**채택 예정**: 4번을 우선 시도. 1·2번을 벤치마크하여 비교 결과를 ADR에 기록.

### 도전 2: 결제-예매 정합성

**문제**: PG 콜백이 늦거나, 중복 도착하거나, 누락될 수 있다. 그럼에도 "결제 성공 = 좌석 확정"은 100% 보장되어야 한다.

**접근**:
- 멱등 키(idempotency key)를 결제 요청 시점에 발급해 PG에 전달.
- 콜백 수신 시 멱등 키로 중복 처리 방지.
- 결제 성공 트랜잭션 안에서 (a) Reservation 상태 변경, (b) Outbox 이벤트 기록을 *함께* 처리.
- 별도 워커가 Outbox를 폴링하여 Kafka로 발행.

### 도전 3: 대기열 공정성

**문제**: FIFO를 보장하면서, 새로고침해도 순번이 유지되어야 한다.

**접근**:
- Redis Sorted Set: score = 진입 timestamp(ms), member = userId.
- 사용자 조회 시 `ZRANK`로 현재 순번 반환.
- 시스템이 일정 주기로 상위 N명을 *팝하지 않고 읽어서* 진입 토큰 발급.
- 토큰은 별도 키(`token:{userId}`)에 TTL과 함께 저장.

### 도전 4: 읽기·쓰기 트래픽 비대칭

**문제**: 좌석 조회는 좌석 점유보다 50배 이상 많다. 같은 모델로 처리하면 조회가 점유를 압박한다.

**접근**:
- 좌석 *상태* 조회는 Redis Hash에서. (Write 쪽이 Redis를 업데이트)
- 좌석 *메타데이터* (구역, 가격) 조회는 애플리케이션 캐시 + MySQL.
- 동일 좌석 배치도에 대한 동시 조회는 Request Collapsing으로 묶음.
- Write Model(MySQL)과 Read Model(Redis)의 일관성은 *최종적 일관성*으로 합의.

### 도전 5: 장애 격리

**문제**: 알림 발송이 느려지거나 통계 적재가 실패해도 핵심 예매는 멈추지 않아야 한다.

**접근**:
- 핵심 경로 (대기열 → 점유 → 결제 → 확정)는 동기.
- 부가 경로 (알림, 통계, 검색 인덱싱)는 Kafka 컨슈머로 분리.
- 한 컨슈머의 장애는 다른 컨슈머에 영향 없음.

| **Spring WebFlux** | Virtual Thread로도 I/O 처리량은 충분. WebFlux의 학습 곡선은 도메인 학습을 방해. |

### 10.2 애플리케이션 내부 구조 — 모듈러 모놀리스

배포 단위는 단일 Spring Boot 애플리케이션이지만, 내부는 도메인 단위 모듈로 엄격히 분리한다.

```
src/main/java/com/project/
├── auth/              ← 인증 모듈
│   ├── api/           ← 외부에 노출되는 인터페이스·DTO
│   ├── domain/        ← 모듈 내부 (외부에서 직접 import 금지)
│   └── infrastructure/
├── waiting/           ← 대기열 모듈
│   ├── api/
│   ├── domain/
│   └── infrastructure/
├── seat/              ← 좌석 점유 모듈
├── reservation/       ← 예매 모듈
├── payment/           ← 결제 모듈 (PG 시뮬레이터 어댑터 포함)
├── notification/      ← 알림 모듈 (Kafka 컨슈머)
├── game/              ← 경기·구장 메타데이터 모듈
└── shared/            ← 공통 값 객체 (UserId, Money 등)
```

**모듈 간 통신 규칙:**

1. **공개 인터페이스 (api 패키지)** — 다른 모듈은 오직 `api` 패키지의 인터페이스·DTO만 import 가능. `domain` 내부 객체는 외부에서 직접 접근 금지.
2. **이벤트 기반 통신 우선** — 가능하면 모듈 간 호출 대신 도메인 이벤트(Spring `ApplicationEvent` 또는 Kafka)로 통신. 결제 확정 → 알림 발송이 대표 케이스.
3. **순환 의존성 금지** — ArchUnit 같은 정적 검사 도구로 빌드 시점에 강제.

**왜 이렇게 하는가:**

- **MSA 분리의 사전 작업**: 모듈 경계는 미래의 *서비스 경계*다. 패키지 단위로 분리해두면, 향후 특정 모듈(예: 결제)을 별도 서비스로 떼어낼 때 가장 큰 비용인 *경계 식별과 의존성 정리*가 이미 끝나 있다.
- **테스트 용이성**: 모듈 단위로 격리된 통합 테스트가 가능하다.
- **인지 부하 관리**: 결제 코드를 볼 때 좌석 점유 코드를 볼 필요가 없다. 모듈 안으로 시야를 좁힐 수 있다.

> 💡 **"왜 처음부터 MSA로 안 하셨어요?" 라는 질문에 대한 입장:** MSA의 본질적 가치는 조직 확장성에 있으며, 1인 프로젝트는 그 이점을 활용할 수 없다. 본 프로젝트는 모듈러 모놀리스로 *MSA의 학습 가치(경계 설계, 이벤트 통신)는 가져가되 운영 비용은 피하는* 의도적 선택이다.

| Kafka/Outbox 구간에서 장기 막힘 | 높음 | 높음 | 1주일 이상 막히면 강사 Q&A·커뮤니티 활용. 그래도 안 풀리면 Outbox 단순화 (워커 → 스케줄러 폴링). |
| AWS 비용 폭탄 | 중 | 높음 | Billing 알람 일 1달러로 설정. 안 쓸 땐 EC2 stop. NAT Gateway 절대 사용 금지. |
| 부하 테스트가 목표 수치 미달 | 높음 | 낮음 | 미달 자체는 실패 아님. *원인 분석과 개선 시도*가 학습 산출물. |
| 6개월차에 인프라+문서를 동시에 못 끝냄 | 중 | 높음 | 5개월차 종료 시점에 코드 freeze. 6개월차는 신규 기능 금지. |
| 면접 시 시연이 안 됨 | 낮음 | 높음 | 면접 1주일 전부터 환경 켜놓고 상시 동작 확인. |

## 부록 A. 용어 정의

| 용어 | 정의 |
|---|---|
| **점유 (Hold)** | 사용자가 좌석을 임시로 잡은 상태. TTL(7분) 후 자동 해제. |
| **확정 (Confirm)** | 결제 성공 후 좌석이 영구 할당된 상태. |
| **진입 토큰 (Entry Token)** | 대기열 통과 후 좌석 선택 페이지에 진입할 권한. TTL(15분). |
| **멱등 키 (Idempotency Key)** | 동일 요청의 중복 처리를 방지하는 식별자. |
| **Outbox** | 트랜잭션 정합성을 위한 이벤트 임시 저장 테이블. |
