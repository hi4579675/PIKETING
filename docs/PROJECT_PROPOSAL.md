## 이 프로젝트의 코어

이 기획안을 딱 세 줄로 줄이면 이렇다.

1. **좌석 동시 점유 정합성** — 한 좌석에 여러 요청이 몰려도 한 명만 통과시킨다.
2. **결제와 예매 상태 정합성** — 동기 버전으로 먼저 만들고, 이중 쓰기 문제가 드러나면 Outbox로 리팩토링한다.
3. **end-to-end 시연 가능한 흐름** — 조회 → 점유 → 결제 → 확정/해제가 한 번에 돌아간다.

나머지 패턴(대기열 풀 스펙, Kafka 컨슈머 분리, CQRS 등)은 위 세 가지를 돕는 보조 역할이다. 일정이 빠듯해지면 *보조부터* 잘라낸다. 평가 피드백을 받고 나서 범위를 다시 잡은 결과다.

## 1. 이게 무슨 프로젝트인가

### 1.1 한 줄 소개

KBO 인기 매치의 티켓 오픈 순간 — 몇 만 명이 1분 안에 몰리는 상황 — 을 견디는 것을 목표로 하는 단일 구단 티켓팅 백엔드 API.

### 1.2 왜 만드는가

신입 취준 중에 분산 시스템 주제를 많이 봤는데, 대부분 *이론으로만* 접했다. 블로그 글, 강의 슬라이드, 책 요약 수준이었다. 직접 손으로 부딪혀본 적은 없었다.

마침 한화 신구장 개장 이후 KBO 티켓 오픈런이 뉴스에 자주 올라오는 걸 보고, "이거다" 싶었다. 실제로 매년 터지는 문제고, 원인을 추적하면 분산 시스템 교과서에 나오는 패턴들이 그대로 등장한다. 내가 학습하려던 주제와 문제 영역이 맞물려 있었다.

그래서 이 프로젝트는 "기술 스택을 연습하는 숙제"가 아니라 **"현실에서 매년 터지는 문제를, 내가 학습 중인 도구로 어디까지 풀 수 있는지 확인해보는 실험"**에 가깝다.

### 1.3 구단·구장 설정

대전 신구장(한화생명 볼파크)을 모티브로 삼았다. 이유 두 가지.

첫째, 2025년 신축 후 시즌 매진 사례가 화제가 된 구단이라 "오픈런"이라는 가정이 뉴스로 검증된다. 면접에서 "왜 한화?"라는 질문에 30초 안에 답할 수 있다.

둘째, 단일 신축 구장이라 좌석 체계가 단순하다. LG/두산 잠실 공유처럼 복잡한 운영 모델이 없어 도메인 모델링 부담이 적다.

본 프로젝트는 학습·포트폴리오 목적의 시뮬레이션이며 실제 구단이나 구장과 아무 연관이 없다. 좌석 배치는 공개 정보를 참고한 가상 데이터다. 구단 로고나 공식 API를 쓰지 않는다.

## 2. 어떤 문제를 풀려는 건가

### 2.1 기존 티켓팅 시스템의 통증

| # | 통증 | 이 프로젝트가 다루는가 |
|---|---|---|
| 1 | 오픈 직후 서버가 죽음 | O |
| 2 | 같은 좌석이 두 명에게 팔림 | O |
| 3 | 결제는 됐는데 좌석 확정이 안 됨 | O |
| 4 | 새로고침하면 대기 순번이 뒤로 밀림 | O (단, 최소 스펙) |
| 5 | 좌석 선택 후 결제 페이지에서 무한 대기 | O |
| 6 | 모바일 UX 불편 | X (스코프 외) |
| 7 | PG사 장애 시 전체 마비 | O (장애 격리) |

### 2.2 기술 문제로 환원

위 통증들을 기술 언어로 바꾸면 이렇게 줄어든다.

**하나, 동시성 정합성.** 단일 자원(좌석)에 대한 동시 요청이 정확히 한 명에게만 허용돼야 한다. 이게 가장 어렵다.

**둘, 두 시스템에 원자적으로 쓸 수 없음.** 결제 DB와 이벤트 발행(Kafka)을 동시에 원자적으로 처리할 방법이 없다. 이걸 Outbox 패턴으로 우회한다.

**셋, 읽기와 쓰기 트래픽 비대칭.** 좌석 조회는 엄청 많고, 점유는 그보다 훨씬 적다. 같은 저장소로 처리하면 조회가 점유를 죽인다.

**넷, 공정성.** 먼저 들어온 사용자가 먼저 처리돼야 한다. 분산 환경에서 완벽한 FIFO는 불가능하므로 "충분히 공정한" 수준을 목표로 한다.

**다섯, 장애 격리.** 알림 처리가 막혔다고 결제가 멈추면 안 된다.

본 프로젝트의 기능들은 이 다섯 문제를 드러내기 위한 수단이다.

## 3. 할 것과 안 할 것

### 3.1 범위를 두 단계로 나눈 이유

기획안 초안(v0.1~v0.3)에서는 모든 걸 P0로 뭉쳤다. 평가 피드백을 받고 다시 보니 현실성이 부족했다. 4개월 풀타임으로도 빡빡한데 초안 수준을 다 구현하면 *어느 하나도 깊지 않게* 될 위험이 컸다.

그래서 v0.4에서는 P0를 **1차 완성본**과 **2차 보강**으로 나눈다. 1차는 "면접에서 시연 가능한 최소한의 완성본", 2차는 "시간이 남으면 리팩토링해서 얹는 층"이다. 이 순서가 학습 면에서도 더 낫다고 판단했는데, 그 이유는 §5.4에서 따로 적는다.

### 3.2 1차 완성본 (반드시 한다)

- 회원가입/로그인/JWT
- 경기 조회, 좌석 배치도 조회
- 좌석 점유 (Redis 원자 점유 + MySQL 유니크 제약의 다층 방어)
- 결제 시뮬레이터 (시나리오 3개: 성공, 실패, 멱등 재요청)
- 결제 → 예매 확정 (단순 동기 처리)
- 점유 7분 TTL 자동 해제
- 예매 내역 조회
- 대기열 최소 버전 (진입, 순번 조회, 진입 토큰 발급)
- 모듈러 모놀리스 + ArchUnit 경계 검증
- 부하 테스트 (MySQL 락 vs Redis 원자 점유 비교)

이게 끝나면 "조회 → 점유 → 결제 → 확정"의 한 줄 시연이 가능해진다.

### 3.3 2차 보강 (1차 끝난 뒤)

- **Outbox 패턴으로 리팩토링** (결제-예매 정합성)
- **Kafka 도입 + 컨슈머 1개** (notification, 로그 출력만)
- Outbox와 멱등 컨슈머 통합 테스트
- AWS 배포 (EC2, RDS, ALB)
- CI/CD (GitHub Actions)

2차 보강의 핵심은 Outbox 리팩토링이다. 이건 §5.4에 따로 이유를 적는다.

### 3.4 시간 정말 남으면 (P1)

- Kafka 컨슈머 추가 (statistics)
- DLT 처리
- 대기열 풀 스펙 (새로고침 안전성 검증, 공격 시나리오 8종 통합 테스트)
- 인기 경기 랭킹

### 3.5 안 하는 것 (의도적으로)

명시적으로 제외한다. 빼먹어서 안 하는 게 아니라, 학습 목표와 거리가 멀거나 완주를 위협한다고 판단해서 뺐다.

- **프론트엔드** — Postman/Swagger로 검증한다. UI는 만들지 않는다.
- **확장 인증** — refresh token rotation, 이메일/본인 인증, 비밀번호 재설정, 소셜 로그인(카카오/네이버), 로그아웃 토큰 블랙리스트, 계정 잠금. 기본 로그인·JWT만 넣는다.
- **실제 PG 연동** — 시뮬레이터로 대체. 다만 PG의 이상 동작(지연, 멱등, 콜백, 실패)은 흉내 낸다.
- **환불, 재결제, 시즌권, 양도** — 도메인 복잡도만 키운다.
- **모바일 푸시, 이메일 실제 발송** — 2차 보강 단계에서도 로그 출력만 한다.
- **다구단 통합** — 단일 구단으로 충분히 핵심 문제를 드러낼 수 있다.
- **CQRS Read 모델 분리** — ERD에서 좌석 상태를 Redis로 뺀 수준까지만 적용한다. 별도 Read DB는 만들지 않는다.
- **Request Collapsing** — 학습 가치는 있지만 본 프로젝트의 트래픽 규모에서 효과 검증이 어렵다.
- **Virtual Thread 성능 검증** — 사용은 하되 별도 실험은 안 한다.
- **Kubernetes, Terraform** — EC2 + Docker Compose로 충분하다.
- **MSA** — 모듈러 모놀리스로 간다. 이유는 ADR-0002에 길게 써뒀다.

비목표를 명시하는 이유가 있다. 신입 포트폴리오의 가장 큰 함정이 "할 수 있을 것 같은데" 하고 스코프가 늘어나는 거다. 비목표가 문서에 박혀 있으면 유혹이 생길 때 꺼내 볼 수 있다. 면접에서 "왜 이건 안 만드셨어요?"라는 질문에도 "빠뜨린 게 아니라 뺀 것"으로 답할 수 있다.

## 4. 누가, 어떻게 쓰는가

### 4.1 상상한 사용자

**김덕후.** 30대 직장인, 한화 시즌권 노림. 오픈 시간에 알람 맞춰두고 대기. 원하는 좌석을 빠르고 공정하게 확보하고 싶어함.

**박입문.** 20대, 친구 따라 첫 야구장. 어디 앉을지 모름. 좌석 등급 정보 보고 결정하고 싶어함.

UI를 만들지 않으므로 이 페르소나는 API 시나리오 검증용으로만 쓴다. 실제 시연은 Postman/Swagger로 한다.

### 4.2 사용자 시나리오

**시나리오 A — 평일 저녁 예매 (저트래픽).**
김덕후가 다음 주 화요일 경기의 좌석 배치도를 조회한다. 1루 외야 자유석을 선택한다. 결제 페이지로 이동한다. 결제를 완료한다. 예매 내역을 확인한다.

**시나리오 B — 오픈런 예매 (고트래픽).** 이게 본 프로젝트의 핵심이다.

한국시리즈 1차전 티켓 오픈 5분 전, 5만 명이 대기 페이지에 접속해 있다. 정각에 대기열이 열린다. 김덕후의 순번이 호출되어 진입 토큰을 받는다. 좌석 선택 페이지에 들어간다. 1루 응원석 A-3-15를 선택하는데, 같은 좌석을 같은 순간 12명이 노리고 있다. 시스템은 정확히 한 명에게만 점유를 준다. 김덕후가 성공하고, 7분 안에 결제한다. 7분 지나면 좌석은 자동 해제돼 다른 사용자에게 돌아간다.

**시나리오 C — 결제 지연.**
김덕후가 결제를 시도했는데 PG 응답이 30초 동안 없다. 시스템은 결제 상태를 "처리 중"으로 유지한다. 좌석도 점유 상태 유지. PG 응답이 도착하면 성공이든 실패든 멱등하게 처리한다.

### 4.3 면접 시연 플로우 (5분 안에)

면접관이 "직접 돌려볼 수 있나요?" 했을 때 보여줄 스크립트. 이걸 미리 짜두지 않으면 현장에서 허둥댄다.

1. **(30초) 환경 확인.** Swagger UI 열기. `/health` 응답 보여주기.
2. **(30초) 회원가입·로그인.** 유저 두 명을 미리 만들어두고, 로그인해서 JWT 받기. 두 개 탭에 각각 JWT 세팅.
3. **(1분) 좌석 배치도 조회.** 특정 경기의 좌석 목록 조회. 이때 쿼리 응답 시간 보여주기. EXPLAIN 결과를 미리 캡처해둔 걸 보여줄 수도 있음.
4. **(1분) 동시 점유 데모.** 두 탭에서 같은 좌석을 동시에 점유 시도. 한 탭은 200, 다른 탭은 409. 로그에서 Lua 스크립트가 어떻게 동작했는지 보여주기.
5. **(1분) 결제 성공 흐름.** 성공한 탭에서 결제 시뮬레이터로 결제. 예매 확정 상태가 DB에 남는 걸 직접 조회해서 보여주기.
6. **(1분) TTL 만료 데모.** 좌석 점유 후 결제 안 하고 7분 기다림... 은 현실적으로 안 되므로, TTL을 10초로 낮춘 시연용 프로파일을 따로 만들어둔다. 10초 후 좌석이 자동 해제되는 걸 보여준다.

이 시나리오에 k6 부하 테스트 결과를 곁들일 수도 있다. 그 경우 부하 테스트는 미리 실행해둔 리포트 화면을 띄운다 — 면접 중에 돌리면 시간 부족.

**준비물:**
- 시드 데이터 (경기 1개, 좌석 2만개)
- 사용자 2명 (kim, park)
- 시연용 `application-demo.yml` (TTL 10초로 단축)
- 부하 테스트 결과 PDF 또는 스크린샷

## 5. 기능 우선순위

### 5.1 1차 완성본 (P0)

| 기능 | 설명 |
|---|---|
| 회원가입 | userId, password 받아서 BCrypt 해싱 후 저장 |
| 로그인 | 자격 검증 후 JWT 발급 (단일 access token, 긴 TTL) |
| JWT 인증 필터 | Spring Security 기반. 보호 엔드포인트 자동 검증 |
| 경기 목록 조회 | 예매 가능 경기 목록 |
| 경기 상세 조회 | 특정 경기 정보 + 좌석 등급별 잔여 요약 |
| 좌석 배치도 조회 | 특정 경기의 전체 좌석 + 상태(가용/점유/확정) |
| 대기열 진입 | 사용자가 대기열 등록, 순번 받음 |
| 대기열 순번 조회 | 현재 순번, 앞에 남은 사람 수 |
| 진입 토큰 발급 | 워커가 순번 따라 토큰 발급 |
| 좌석 점유 | 진입 토큰 검증 후 Redis 원자 점유 |
| 결제 요청 | 점유한 좌석에 대해 결제 시작 |
| 결제 시뮬레이터 | SUCCESS / FAILURE / IDEMPOTENT_REPLAY 3종 |
| 결제 성공 처리 | 예매 확정, 좌석 점유 해제 (동기 트랜잭션) |
| 점유 TTL 자동 만료 | 7분 후 자동 반환 |
| 예매 내역 조회 | 본인 예매 내역 |
| 예매 취소 | 본인 예매 취소 |

### 5.2 2차 보강 (P0.5)

| 기능 | 설명 |
|---|---|
| Outbox 테이블 도입 | 결제 트랜잭션에 outbox 이벤트 INSERT 포함 |
| Outbox 워커 | SKIP LOCKED로 폴링, Kafka 발행, 재시도 |
| 결제 리팩토링 | 동기 Kafka 호출 제거, Outbox 기반 비동기로 전환 |
| notification 컨슈머 | 단일 Kafka 컨슈머, 멱등 처리, 로그 출력 |
| 통합 테스트 | Outbox 발행·컨슈머 처리·멱등성 검증 |
| AWS 배포 | EC2 2대 + RDS + ALB + HTTPS |
| CI/CD | GitHub Actions, main 푸시 시 자동 배포 |

### 5.3 P1 (시간 남으면)

| 기능 | 설명 |
|---|---|
| statistics 컨슈머 | 매출·예매 통계 적재. 컨슈머 그룹 분리 실습 |
| DLT 처리 | 영구 실패 메시지 격리 |
| 대기열 풀 스펙 | 공격 시나리오 8종 검증 |
| 인기 경기 랭킹 | 일별 점유 시도 수 기반 |
| 관리자 기능 | 경기 등록, 좌석 등록 |

### 5.4 왜 Outbox를 2차로 미뤘는가

이 순서 결정이 v0.4의 가장 중요한 변경이다. 초안에서는 Outbox를 P0에 넣었다.

평가 피드백을 보고 다시 생각해봤는데, **"동기로 먼저 짜고 Outbox로 리팩토링"이라는 순서 자체가 학습 산출물**이 될 수 있겠다 싶었다.

이유는 이렇다. 동기 버전을 먼저 짜면, 통합 테스트를 돌리다가 자연스럽게 이중 쓰기 문제를 발견하게 된다. "Kafka 잠시 죽여놓고 결제해보니 DB는 커밋됐는데 이벤트가 사라졌네" 같은 순간이 온다. 그 순간이 바로 *Outbox를 쓰는 이유*를 내가 직접 체감하는 순간이다. 처음부터 Outbox로 짜면 이 체감이 없다. "교과서에 그렇게 하라고 해서 했습니다" 수준에 머문다.

면접 답변 입장에서도 차이가 크다.

초안 방식 답변: "Outbox 패턴을 써서 결제와 이벤트 발행을 원자적으로 처리했습니다."
v0.4 방식 답변: "처음엔 결제 트랜잭션 안에서 Kafka send를 직접 호출했습니다. 통합 테스트에서 Kafka를 중단시킨 상태로 결제해봤더니 DB에는 결제 성공이 찍혔는데 이벤트는 사라지는 케이스를 발견했습니다. 이게 dual write 문제라는 걸 알고 Outbox로 리팩토링했습니다."

두 번째가 훨씬 강하다. 실제로 경험한 것처럼 보이기 때문이 아니라, 실제로 경험했기 때문에 그렇게 말할 수 있게 되는 것이다.

리팩토링 비용은 걱정했던 것보다 작다. Outbox는 결제 서비스의 *한 메서드*만 바꾸는 작업이다. 도메인 전체 구조를 뒤엎는 변경이 아니다. 2차 보강의 1~2주면 끝낼 수 있다고 본다.

물론 위험은 있다. 1차 완성본에서 시간을 다 써버리면 2차 보강에 진입 못 할 수 있다. 그 경우 "Outbox는 설계 문서에만 있고 코드에는 없음" 상태가 된다. 이건 마이너스다. 그래서 1차 완성본 스케줄을 빡빡하게 잡지 말고, 3.5개월 안에 끝내는 걸 목표로 해야 한다.

## 6. 비기능 요구사항

숫자는 목표고, 측정 없이 고정하지 않는다. 부하 테스트로 검증하고 못 맞춘 경우 "왜 못 맞췄는지"를 분석하는 것 자체가 학습 산출물이다.

### 6.1 수치를 정한 근거

본 프로젝트가 가정하는 트래픽은 이렇다.

- 동시 대기 사용자 최대 5만 명 (한화 신구장 약 2만 좌석의 2.5배 — 인기 매치에서 좌석 수 초과 접속은 흔함)
- 오픈 직후 1분 동안 대기열 진입 요청이 약 5천 req/s로 몰림
- 좌석 배치도 조회는 대기 통과 후 폴링 + 선택 화면에서 발생, 초당 수천 건 수준
- 실제 좌석 점유 시도는 그보다 훨씬 적음, 초당 수백 건

이 가정을 바탕으로 아래 목표를 잡았다. 가정 자체가 단일 구단 기준 추정치고, 한화 외 구단이나 프로야구 외 종목에는 해당 안 된다.

### 6.2 성능 목표

| 지표 | 목표 | 측정 방법 |
|---|---|---|
| 좌석 조회 처리량 | 3,000 req/s 이상 | k6 |
| 좌석 점유 처리량 | 500 req/s 이상 | k6 |
| 좌석 조회 p99 | 200ms 이하 | k6 + Actuator |
| 좌석 점유 p99 | 500ms 이하 | k6 + Actuator |
| 대기열 진입 처리량 | 5,000 req/s 이상 | k6 |

### 6.3 정합성 목표

| 지표 | 목표 |
|---|---|
| 동일 좌석 중복 점유 | 0건 (10만 건 부하 테스트 기준) |
| 결제 성공 후 좌석 미확정 | 0건 |
| 결제 실패 후 좌석 미반환 | 0건 |
| 대기열 FIFO 위반 | 5% 이내 (같은 ms 내 미세 역전 허용) |

### 6.4 가용성 한계

- 알림 시스템 장애 시: 결제 플로우 정상 (2차 보강 이후)
- Redis 단일 장애 시: 좌석 점유 일시 중단, 데이터 손실 없음 (휘발성이므로)
- Kafka 장애 시: 결제는 정상, 이벤트 발행은 복구 후 재처리 (2차 보강 이후)
- AWS RDS 장애 시: 서비스 중단. 본 프로젝트는 단일 DB 가정.

### 6.5 수치 미달 시

수치를 못 맞추는 건 실패가 아니다. 못 맞춘 원인을 분석해서 README와 부하 테스트 리포트에 기록하는 것이 학습의 일부다. 예를 들어 "500 req/s 목표였는데 320에서 멈춤, EXPLAIN 분석 결과 `game_game_seats.idx_game`이 예상대로 동작했지만 Lua 스크립트의 Redis 라운드트립이 병목" 같은 분석이 있으면 오히려 수치 달성보다 면접 답변이 강해진다.

## 7. 도메인 모델 요약

상세는 `docs/ERD.md`에. 여기서는 핵심 엔티티와 설계 판단만 짧게 적는다.

### 7.1 주요 엔티티

- **Stadium** — 구장. 단일 인스턴스.
- **Section** — 구역. 좌석 등급의 단위.
- **Seat** — 물리 좌석.
- **Game** — 경기.
- **GameSeat** — 경기별 좌석 인스턴스. Game × Seat 조합에 가격 정보.
- **Reservation** — 예매. 결제 성공 시 만들어진다.
- **Payment** — 결제 트랜잭션.
- **OutboxEvent** — 2차 보강 단계에서 추가. 결제 확정 이벤트의 임시 저장소.
- **WaitingEntry** — 대기열 항목 (Redis에만 존재, RDB 테이블 없음).

### 7.2 Seat과 GameSeat을 분리한 이유

물리 좌석은 바뀌지 않지만 경기마다 가격·상태가 다르다. 합쳐두면 한 좌석이 여러 경기에 동시 존재해야 하는 상황을 모델링할 수 없다.

### 7.3 Reservation과 Payment를 분리한 이유

Reservation은 "좌석 잡았다"는 사실, Payment는 "돈이 오갔다"는 사실이다. 둘은 수명이 다르다. 예를 들어 결제는 부분 환불·재시도가 가능하지만 예매는 그와 별개다. 당장 본 프로젝트에 부분 환불은 없지만, 합쳐두면 나중에 확장이 막힌다.

### 7.4 좌석 상태를 DB에 두지 않는 이유

좌석의 *현재 점유 상태*는 초당 수천~수만 회 바뀔 수 있다. MySQL에 박으면 락 경합이 폭발한다. 그래서 상태는 Redis에만 둔다. ERD에 `seat_status` 같은 컬럼이 없는 이유다.

"영구 할당됐는가"는 `rsv_reservations` 테이블의 존재 여부로 판단한다. 즉 두 저장소가 각자 다른 의미의 상태를 담는다. 이게 본 프로젝트의 읽기·쓰기 분리의 부분 적용이다.

## 8. 기술적 도전 과제

이 섹션이 면접에서 가장 많이 질문받을 영역이다. 설계 문서들이 이 5가지를 각각 깊게 다룬다.

### 도전 1. 좌석 동시 점유

**상황.** 한 좌석에 수십~수백 요청이 동시에 몰린다. 정확히 한 명만 통과시켜야 한다.

**접근.** 비교할 세 가지 방법을 세웠다.

1. **MySQL 비관적 락** — `SELECT ... FOR UPDATE`. 단순하지만 락 경합 시 처리량이 떨어진다. baseline으로 삼는다.
2. **MySQL 낙관적 락** — version 컬럼. 충돌 시 재시도 비용. 인기 좌석에선 대부분 실패한다.
3. **Redis 원자 점유** — Lua 스크립트로 EXISTS + HSET을 원자 처리. 락이 아니라 *상태 자체*를 한 연산에 점유한다.

1차 완성본에서 세 가지 다 구현해 부하 테스트로 비교한다. 비교 지표는 처리량(req/s), p99 응답시간, 중복 점유 발생률(0이어야 함). 예상으로는 Redis 원자 점유가 가장 좋을 것 같지만, 측정해봐야 안다.

**다층 방어.** Redis 점유(1층) + MySQL CONFIRMED 사전 검증(2층) + MySQL `uk_active_seat` 유니크 제약(3층). 1층은 처리량 최적화, 3층은 정합성 최후 보루.

→ `docs/SEAT_HOLD_DESIGN.md`

### 도전 2. 결제와 예매 상태 정합성

**상황.** 결제 성공 → 예매 확정 → 알림 발행의 세 단계를 한 트랜잭션으로 묶고 싶다. 그런데 DB와 Kafka는 서로 다른 시스템이라 동시에 원자적으로 쓸 수 없다.

**접근.**

- **1차 완성본**: 그냥 동기로 처리한다. 결제 성공 시 `@Transactional` 메서드 안에서 payment UPDATE + reservation INSERT. Kafka는 아예 안 쓴다.
- **2차 보강**: 통합 테스트에서 이중 쓰기 문제를 드러낸 뒤, Outbox 패턴으로 리팩토링. 결제 트랜잭션 안에서 outbox 테이블에 이벤트 INSERT → 별도 워커가 폴링해서 Kafka 발행.

1차 → 2차 순서로 가는 이유는 §5.4에서 길게 적었다.

→ `docs/OUTBOX_PATTERN_DESIGN.md`, `docs/PAYMENT_SIMULATOR_DESIGN.md`

### 도전 3. 대기열 공정성

**상황.** FIFO를 보장하면서 새로고침해도 순번이 밀리지 않아야 한다.

**접근.** Redis Sorted Set을 사용한다. score는 진입 timestamp, member는 userId. 핵심 디테일은 `ZADD NX` 옵션으로, 이미 존재하는 사용자의 score는 덮어쓰지 않는다. 이게 빠지면 새로고침 = 순번 밀림 버그가 박힌다.

진입 토큰은 대기열과 분리된 별개 자원이다. Redis String + TTL 15분. 좌석 점유 API는 이 토큰을 헤더로 받아 검증한다. 토큰 없이는 좌석 API 자체가 401.

1차 완성본에서는 대기열의 최소 기능만 구현한다. 완전한 새로고침 안전성 검증과 공격 시나리오 8종은 P1로 미룬다.

→ `docs/WAITING_QUEUE_DESIGN.md`

### 도전 4. 읽기와 쓰기 트래픽 비대칭

**상황.** 좌석 배치도 조회는 초당 수천 건, 점유 시도는 그보다 훨씬 적다. 같은 저장소로 처리하면 조회가 점유를 압박한다.

**접근.**

- 좌석 *영구 정보* (구역, 가격): MySQL
- 좌석 *현재 점유 상태*: Redis
- 좌석 *영구 할당*: MySQL `rsv_reservations`

좌석 배치도 조회는 MySQL에서 경기별 전체 GameSeat을 뽑고, Redis에서 현재 점유 상태를 조회해 합친다. 이게 CQRS 사상의 부분 적용이다. 별도 Read DB는 만들지 않는다.

→ `docs/ERD.md §1.3`

### 도전 5. 장애 격리

**상황.** 알림 처리 지연이 결제에 영향을 주면 안 된다.

**접근.** 2차 보강 단계에서 실현한다. 결제 확정과 알림 발행을 Outbox + Kafka로 분리해, 알림 컨슈머가 죽거나 느려져도 결제 플로우는 영향을 받지 않는다.

1차 완성본에서는 장애 격리가 달성되지 않는다. 결제와 알림이 같은 트랜잭션에 있기 때문이다. 2차 보강이 이 문제를 푸는 이유가 여기 있다.

→ `docs/KAFKA_CONSUMER_DESIGN.md`

## 9. 기술 스택과 선정 근거

### 9.1 언어·프레임워크

- **Java 21** — 현재 LTS. Virtual Thread가 나와서 I/O 바운드 처리 효율에 도움. 별도 실험은 비목표지만 기본적으로 쓴다.
- **Spring Boot 3.x** — 한국 백엔드 시장의 사실상 표준. 강의 자료와 일치.
- **Spring Security + JWT** — Stateless. 본 프로젝트가 앱 서버 2대 분산 구조라 세션 스토어 없이 인증이 굴러간다는 점에서 유리. 인증 자체는 학습 주제가 아니므로 최소 스펙만.
- **Spring Data JPA** — 도메인 엔티티 매핑 기본. Native Query는 필요할 때 병행.

### 9.2 데이터 저장소

- **MySQL 8** — 한국 시장 표준. 강의가 MySQL 기준이라 학습 자료 정합. MySQL 8의 SKIP LOCKED는 Outbox 워커에 중요하게 쓴다. 자세한 트레이드오프는 ADR-0004에.
- **Redis 7** — 좌석 점유 상태, 대기열 순서, 진입 토큰. 세 가지 역할을 자료구조를 달리해 처리한다 (Hash, Sorted Set, String).
- **Kafka** — 2차 보강 단계에서 도입. 결제 후 비동기 파이프라인, 장애 격리의 본체.

### 9.3 인프라·운영

- **Docker Compose** — 로컬 개발 환경. MySQL, Redis, Kafka를 한 번에.
- **AWS EC2** — 앱 서버 2대. ECS/EKS는 학습 곡선 대비 본 프로젝트 일정에 맞지 않음.
- **AWS ALB** — 앱 서버 트래픽 분산. HTTPS 종료.
- **AWS RDS for MySQL** — 백업·복구·장애조치를 관리형으로 위임. 학습 자원을 애플리케이션 레이어에 집중하기 위한 선택.
- **Redis, Kafka는 EC2 위에 직접 (Docker)** — ElastiCache, MSK는 비용 부담. 직접 운영하면서 동작 원리를 손에 익히는 게 학습 목표에 부합.
- **GitHub Actions** — 무료, 레포 안에 워크플로 yml이 같이 있음.
- **k6** — 부하 테스트. JS로 시나리오 작성, 결과를 코드로 관리 가능.
- **ArchUnit** — 모듈 경계 빌드 시점 강제.

### 9.4 의도적으로 안 쓰는 것

| 기술 | 안 쓰는 이유 |
|---|---|
| Kubernetes (EKS) | 6개월 일정에 무리. 얕게 쓰면 면접에서 역효과. 다음 프로젝트에서. |
| Terraform | 인프라 코드화의 학습 가치는 크지만 본 프로젝트의 핵심은 애플리케이션 레이어. 콘솔로 구성하고 결과를 문서화. |
| MSA | 단일 도메인 1인 프로젝트에서 MSA의 이점(조직 확장성)을 누릴 수 없음. ADR-0002에 자세히. |
| Spring WebFlux | Virtual Thread로 충분. WebFlux는 학습 곡선이 본 프로젝트의 주제를 방해한다. |
| MariaDB | MySQL과 거의 동일하나 한국 시장 점유율이 낮음. 쓸 이유 없음. |

## 10. 아키텍처

### 10.1 배포 구조

```
                                  ┌─────────────┐
                                  │   Client    │ (Postman/Swagger)
                                  └──────┬──────┘
                                         │ HTTPS
                                  ┌──────▼──────┐
                                  │     ALB     │
                                  └──────┬──────┘
                                         │
                          ┌──────────────┴──────────────┐
                  ┌───────▼───────┐             ┌───────▼───────┐
                  │  Spring Boot  │             │  Spring Boot  │
                  │   (EC2 #1)    │             │   (EC2 #2)    │
                  └───┬───────┬───┘             └───┬───────┬───┘
                      │       │                     │       │
        ┌─────────────┘       └────────┬────────────┘       │
        │                              │                    │
   ┌────▼────┐               ┌─────────▼──────┐      ┌──────▼──────┐
   │   RDS   │               │  Redis (EC2)   │      │ Kafka (EC2) │
   │ (MySQL) │               │ - 좌석 상태    │      │ (2차 보강)  │
   │         │               │ - 대기열       │      └─────────────┘
   │         │               │ - 진입 토큰    │
   └─────────┘               └────────────────┘
```

Kafka 칸은 2차 보강 단계에서 실제로 쓰이기 시작한다. 1차 완성본 시점에서는 컨테이너만 떠 있고 아무도 사용하지 않는다.

### 10.2 내부 구조 — 모듈러 모놀리스

단일 Spring Boot 애플리케이션이지만 내부는 도메인별로 분리한다.

```
src/main/java/com/project/
├── auth/
│   ├── api/
│   ├── domain/
│   └── infrastructure/
├── game/
├── waiting/
├── seat/
├── reservation/
├── payment/
├── notification/       (2차 보강부터 실제 사용)
└── shared/
```

**규칙.**

- 모듈 간 직접 의존 금지. 공개 인터페이스(`api` 패키지) 또는 이벤트로만 통신.
- `domain` 패키지는 외부에서 import 금지.
- 순환 의존 금지.

이 규칙들을 ArchUnit 테스트로 빌드 시점에 검증한다. 어긴 커밋은 CI에서 막힌다.

MSA로 가지 않은 이유와 모듈 경계 설계의 배경은 ADR-0002에 자세히 적었다. 여기서는 규칙 자체만 다룬다.

| 리스크 | 가능성 | 영향 | 대응 |
|---|---|---|---|
| 강의 진도가 계획보다 늦어짐 | 높음 | 중 | P1 기능 컷. 1차 완성본만 사수. |
| Outbox 리팩토링에서 장기 막힘 | 중 | 높음 | 1주일 이상 막히면 2차 보강을 단순화 (Outbox 워커를 스케줄러 1개로, 재시도 로직 단순화) |
| AWS 비용 폭탄 | 중 | 높음 | Billing 알람 일 1달러. 안 쓸 때 EC2 stop. NAT Gateway 사용 금지. |
| 부하 테스트 수치 미달 | 높음 | 낮음 | 미달 자체는 실패 아님. 원인 분석 + 개선 시도가 학습 산출물. |
| 번아웃 | 중 | 높음 | 주 1일 반드시 휴식. 매주 회고에서 멘탈 상태 체크. |

| 1차 완성본이 3.5개월 초과 | 중 | 높음 | P0 기능 중 대기열 진입 토큰 발급 워커를 수동 호출로 단순화 등 |

**슬립 시 컷 순서.** 일정이 늦어질 때 어떤 순서로 뺄지 미리 정해둔다.

1순위 컷 — 인기 경기 랭킹, 좌석 추천, Prometheus/Grafana, ADR 추가 작성
2순위 컷 — DLT, 컨슈머 그룹 분리 (statistics 제거), 부하 테스트 풀 시나리오
3순위 컷 — AWS 배포 (로컬 Docker만), HTTPS (HTTP만)

**절대 안 자르는 것.** 좌석 원자 점유, 다층 방어, 결제 시뮬레이터, 1차 완성본의 end-to-end 흐름, 부하 테스트 결과 문서화, README와 설계 문서 v0.2 이상 갱신. 이걸 자르면 프로젝트의 정체성이 무너진다.

## 부록 A. 용어 정리

| 용어 | 의미 |
|---|---|
| 점유 (Hold) | 좌석을 임시로 잡은 상태. TTL 7분. Redis에만 존재. |
| 확정 (Confirm) | 결제 성공 후 좌석이 영구 할당된 상태. MySQL에 기록. |
| 진입 토큰 (Admission Token) | 대기열 통과 증명. 좌석 점유 API의 필수 헤더. TTL 15분. |
| 멱등 키 | 중복 처리를 막기 위한 식별자. 결제 요청, 콜백 처리에 사용. |
| Outbox | 결제 트랜잭션과 함께 이벤트를 저장하는 임시 테이블. 2차 보강부터 도입. |
| 다층 방어 | 좌석 점유 정합성을 위한 3층 구조 (Redis 원자 / MySQL 사전 검증 / MySQL 유니크 제약). |
