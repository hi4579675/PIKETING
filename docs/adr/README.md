# Architecture Decision Records (ADR)

이 디렉토리는 본 프로젝트의 주요 기술 의사결정을 기록한다.

## ADR이란?

**"중요한 기술 의사결정을, 결정 시점의 맥락과 함께 한 페이지로 남기는 문서."**

코드만 보면 *무엇을* 했는지는 알 수 있지만, *왜* 그렇게 했는지는 알 수 없다. 
ADR은 결정 시점의 맥락(제약, 대안, 트레이드오프)을 기록하여 미래의 본인·동료·면접관이 그 질문에 답할 수 있게 한다.

## 작성 원칙

- **한 결정 = 한 파일.** 여러 결정을 한 문서에 묶지 않는다.
- **결정 시점에 작성한다.** 사후 정당화는 ADR이 아니다.
- **간결하게.** 한 ADR은 1~2 페이지를 넘기지 않는다.
- **불변(immutable).** 한 번 작성된 ADR은 수정하지 않는다. 
- 결정이 바뀌면 *새 ADR*을 작성하고 이전 ADR의 상태를 `Superseded by ADR-XXX`로 표시한다.
- **번호는 3자리.** 001, 002, ... 순서대로.

## 상태(Status) 정의

| 상태 | 의미 |
|---|---|
| Proposed | 제안됨, 아직 확정 전 |
| Accepted | 채택됨, 현재 유효 |
| Deprecated | 더 이상 권장하지 않음 (대체 ADR 없음) |
| Superseded by ADR-XXXX | 다른 ADR에 의해 대체됨 |

## 작성 시점

다음 중 하나에 해당하면 ADR을 작성한다:

- 되돌리기 어려운 결정 (DB 스키마, 핵심 라이브러리 선택, 아키텍처 스타일)
- 여러 합리적 대안이 있는 결정
- 결정의 *근거*를 6개월 후에도 기억해야 할 결정
- 기존 결정을 *변경*하는 결정

> 💡 의심스러우면 작성한다. ADR은 작성 비용보다 *없을 때의 비용*이 훨씬 크다.
