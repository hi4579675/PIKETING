# ADR-006: 대기열 설계 — Redis Sorted Set + Cursor 기반 Worker 채택

- **상태(Status)**: Accepted
- **작성일**: 2026-04-14
- **결정자**: hanna
- **관련 문서**: PROJECT_PROPOSAL §3.2, §5.1, §6.2, §8

---

## Context (배경)

한국시리즈급 인기 경기 티켓 오픈 시, 5만 명이 동시에 접근한다고 가정한다.
대기열 없이 좌석 점유 API를 직접 노출하면 세 가지 문제가 동시에 터진다.

1. **락 경합 폭발** — 좌석 점유 로직이 직렬화되면서 처리량이 0에 수렴한다.
2. **불공정** — 빠른 네트워크를 가진 사람이 늦게 클릭해도 먼저 좌석을 잡는다.
3. **새로고침 순번 밀림** — 실제 티켓팅 사이트에서 반복 재현되는 문제.

대기열은 이 세 문제를 동시에 해결하는 레이어다.
"5만 명을 줄 세우고, 일정량씩만 좌석 선택 레이어로 흘려보낸다."

제약 조건:
- 1인 개발자, Redis + MySQL 스택 고정 (ADR-001, 인프라 결정)
- 대기열 진입 처리량 목표: 5,000 req/s (PROJECT_PROPOSAL §6.2)
- 새로고침해도 순번이 유지되어야 함 (공정성 요구사항)
- 단일 Redis 인스턴스 (고가용성 구성은 스코프 외)

---

## Alternatives Considered (고려한 대안)

### 대안 A: DB 기반 대기열 (MySQL 테이블)

- **장점**: 트랜잭션·영속성 보장. 별도 인프라 불필요.
- **단점**: 5,000 req/s INSERT에서 락 경합 폭발.
  대기열은 휘발성 데이터라 영속성이 오히려 부담이 됨.
  목표 처리량(5,000 req/s) 달성 불가.

### 대안 B: 메시지 큐(Kafka/RabbitMQ) 기반 대기열

- **장점**: 대용량 처리에 최적화. 순서 보장(파티션 단위).
- **단점**: "현재 내 순번이 몇 번인가" 조회가 불가능 (MQ는 push 모델).
  사용자 폴링 기반 순번 조회를 구현하려면 별도 상태 저장소가 결국 필요함.
  결국 Redis를 써야 하는데 Kafka까지 더하면 복잡도만 증가.

### 대안 C: Redis List (LPUSH/RPOP)

- **장점**: 단순한 FIFO 큐.
- **단점**: "내 순번이 몇 번인가" O(1) 조회 불가 (LPOS는 O(N)).
  새로고침 시 중복 진입 방어 로직을 별도로 구현해야 함.
  Sorted Set이 이 기능을 기본으로 제공하므로 선택 이유 없음.

### 대안 D: Redis Sorted Set + ZPOPMIN (팝 방식 Worker)

- **장점**: Worker 구현이 단순. 꺼내면서 처리하므로 상태 추적 불필요.
- **단점**: 팝 후 토큰 발급 전에 Worker가 죽으면 해당 사용자 데이터 유실.
  재시도 불가 — 데이터가 이미 큐에서 사라졌으므로.

---

## Decision (결정)

**Redis Sorted Set + Cursor 기반 Worker를 채택한다.**

- 대기열 본체: `waiting:queue:{gameId}` (Sorted Set, score = 진입 timestamp ms)
- 사용자 메타: `waiting:user:{gameId}:{userId}` (Hash, status/entered_at/admitted_at)
- Worker 커서: `waiting:cursor:{gameId}` (String, 어디까지 처리했는지 책갈피)
- 진입 토큰: `admission:token:{token}` (String TTL 15분)
- 역방향 매핑: `admission:user:{gameId}:{userId}` (String TTL 15분, 중복 발급 방지)

Worker는 ZPOPMIN 대신 `ZRANGE(cursor, cursor+N-1)` 로 읽고,
성공 후 `INCRBY cursor N` 으로 커서를 전진한다.
실패 시 커서가 움직이지 않아 다음 사이클에 자동 재시도된다.

---

## Rationale (근거)

**Sorted Set을 선택한 이유 — score가 FIFO와 순번 조회를 동시에 해결한다.**
score = 진입 timestamp이므로, 먼저 들어온 사람이 낮은 score를 가진다.
`ZRANK`는 O(log N)으로 현재 순번을 반환한다.
List나 MQ로는 이 두 기능을 O(1)~O(log N)으로 동시에 제공할 수 없다.

**ZADD NX 한 글자가 공정성 전체를 떠받친다.**
`NX` 옵션은 이미 존재하는 member의 score를 덮어쓰지 않는다.
새로고침해서 재요청해도 처음 진입 시각이 그대로 유지된다.
이 한 글자가 빠지면 새로고침할수록 순번이 뒤로 밀리는 현상이 재현된다.

**팝 대신 커서 방식을 선택한 이유 — 재시도 안전성.**
`ZPOPMIN`은 꺼내는 순간 데이터가 사라진다.
처리 실패 시 복구 불가. 커서 방식은 읽기만 하고 커서를 전진하므로,
실패해도 다음 사이클에 같은 범위를 다시 처리할 수 있다.

**토큰과 대기열을 분리한 이유 — 두 관심사의 수명이 다르다.**
대기열은 경기 오픈부터 매진까지 존재한다 (TTL 없음).
토큰은 발급 후 15분만 유효하다 (좌석 선택 기한).
좌석 점유 TTL(7분)과도 의도적으로 다르다. 7분 안에 결제 못 해도
토큰이 살아있으면 다른 좌석을 다시 시도할 수 있다.
이 두 수명을 하나로 합치면 "좌석 하나 놓쳤다고 대기열을 처음부터 다시 타야 하는" 문제가 생긴다.

## Consequences (결과·트레이드오프)

### Positive

- 대기열 진입 처리량 목표(5,000 req/s) 달성 가능.
  Redis 단일 명령(`ZADD NX`)이므로 병목이 Redis 자체 처리량.
- 새로고침 어뷰징 차단. ZADD NX로 Redis 레벨에서 원천 차단.
- Worker 장애 복구 자동화. 커서 전진 실패 = 다음 사이클 재처리.
- 토큰 1회성 보장. `admission:token` 키를 좌석 점유 시 즉시 삭제.
  같은 토큰으로 여러 좌석 점유 경로가 없음.

### Negative

- **FIFO 100% 불가.** 동일 ms 진입자는 score가 같아 사전순 정렬된다.
  네트워크 지연으로 먼저 클릭해도 나중에 도착할 수 있다.
  목표는 "FIFO 위반 5% 이내"이며, 100% 보장은 의도적으로 포기한다.
  100% FIFO는 단일 노드 단일 스레드 직렬 처리를 요구하는데, 이는 처리량 목표와 양립 불가.

- **Redis 단일 인스턴스 SPOF.** Redis 장애 시 대기열 전체가 정지한다.
  대기열 데이터는 휘발성이므로 복구 후 재진입으로 해소 가능하지만,
  기존 순번은 보존되지 않는다.

- **활성 gameId 관리 필요.** Worker가 현재 처리할 gameId 목록을 알아야 한다.
  초기 구현은 하드코딩. game 모듈과의 연동으로 교체 예정.

### Neutral / 향후 영향

- 대기열 데이터가 Redis에만 존재하므로, MySQL에는 대기열 테이블이 없다.
  이는 의도된 설계로, `WaitingEntry` 엔티티를 만들지 않는다.
- 토큰 검증 로직은 seat 모듈이 담당한다.
  waiting 모듈은 토큰 발급까지만 책임지고, 검증 이후는 seat 모듈의 책임이다.

## Revisit Triggers (재검토 조건)

- Redis 단일 인스턴스 장애가 실제로 발생하면 → Sentinel/Cluster 전환 검토
- 동시 접속자가 10만 명을 초과하면 → Worker 분산 전략 재검토
- FIFO 위반율이 5%를 넘는 것이 부하 테스트에서 측정되면 → score 정밀도 향상(ns 단위) 또는 sequence 기반 전환 검토
